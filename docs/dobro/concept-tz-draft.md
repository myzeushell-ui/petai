# «Добро» — голосовой ИИ-компаньон для пенсионеров
## Предварительная концепция и черновое ТЗ (по итогам штурма)

Дата штурма: 2026-07-04. Статус: черновик, часть решений зафиксирована, часть — открытые вопросы.

---

## 1. Идея и миссия

Голосовой ИИ-компаньон в Telegram для аудитории 50+/пенсионеров. Закрывает три
связанные потребности:

1. **Общение и забота** — подтверждение ценности жизненного опыта, ощущение, что
   пенсия «не конец», регулярный тёплый контакт.
2. **Реальная польза для здоровья и быта** — напоминания о лекарствах, разбор
   рецептов/документов, советы по саду/огороду и т.п.
3. **Доступ к возможностям ИИ**, которые обычно недоступны малограмотным в
   компьютерном плане людям — просто сфотографировать что-то и получить голосовой
   ответ, инструкцию, картинку.

Бот не просто отвечает — он **сам проявляет инициативу** (чек-ины, напоминания,
поздравления), т.е. ведёт себя как заботливый компаньон, а не как сервис на вызов.

---

## 2. Целевая аудитория

Люди 50+, пенсионеры, часто с низкой цифровой грамотностью. Отсюда ключевое
требование ко всему продукту: **максимальная простота использования**, минимум
шагов, ставку делаем на голос и фото вместо текста и меню.

---

## 3. Канал и базовая архитектура

- **Канал**: Telegram, голосовые сообщения (входящие и исходящие) + фото.
- **Инициатива**: двусторонняя — пользователь может написать/наговорить в любой
  момент, бот также сам пишет по расписанию/событиям (проактивность).
- **Голосовые технологии**: платные облачные API (STT + TTS + LLM). Конкретный
  вендор — открытый вопрос (см. раздел 8): кандидаты — Yandex SpeechKit / Sber
  SaluteSpeech (лучше для русской речи пожилых людей, распознавание акцентов и
  дикции) vs. Whisper/ElevenLabs/OpenAI (быстрее внедрить, качество TTS выше).

---

## 4. Функциональные блоки

### 4.1 Голосовой диалог
STT → LLM (диалог + логика компаньона) → TTS. Тон ответа — уважительный, тёплый,
без снисходительности, без сложного технического языка.

### 4.2 Память о пользователе
- **Базовый профиль**: имя, возраст, интересы/хобби, ключевые факты о жизни.
- **Расширенно**: история здоровья/лекарств (чувствительные данные — нужны явные
  меры по приватности и согласию, см. п.7).
- **Дневник эмоций/событий** (опционально): что произошло, как себя чувствовал —
  для более «человечных» и уместных ответов со временем.
- Новые факты, всплывающие в разговоре, запоминаются автоматически и используются
  в дальнейших ответах и планировании (напоминания, темы для чек-инов).

### 4.3 Проактивность (бот пишет первым)
Зафиксированный на старте набор:
- Напоминания о приёме лекарств/режиме (по расписанию или по данным из
  распознанного рецепта).
- Утренний/вечерний чек-ин «как дела».
- Поздравления с датами (день рождения, праздники, годовщины).
- Детект «давно нет на связи» → бот сам пишет пользователю и/или предупреждает
  родных (точная логика — открытый вопрос, см. п.8).

### 4.4 Мультимодальные фото-сценарии
Пользователь фотографирует — бот распознаёт и отвечает голосом/текстом/картинкой:
- Рецепты и медицинские документы → расшифровка, разъяснение, как принимать
  (обязательно с дисклеймером «не заменяет консультацию врача»).
- Растения/вредители, уход за садом и огородом → распознавание + советы по уходу
  и выращиванию.
- Бытовые документы (квитанции ЖКХ, письма из госучреждений) → простое объяснение,
  что это и что делать.
- Еда/продукты → расшифровка состава, польза/вред.
- **Сегментация по увлечениям**: для разных категорий пользователей — разные
  «ветки» экспертизы (например, для рыбаков — снасти/рыбалка/погода, для
  пчеловодов — пасека и т.п.). Список хобби-веток будет расти по мере пилота.

### 4.5 Генерация фото/видео
Бот может дополнять голосовые ответы и инструкции сгенерированными изображениями
или короткими видео (например, наглядная пошаговая инструкция, иллюстрация к
совету по уходу за растением, поздравительная открытка).

### 4.6 Профиль интересов и онбординг
Комбинация трёх способов:
1. Визуальный способ (по аналогии с Pinterest) — пользователю показывают 3–4
   изображения по теме, он «кликает»/выбирает то, что откликается, бот подстраивает
   контент под него. Хорошо подходит для аудитории, которой сложно формулировать
   текстом.
2. Явная короткая анкета при первом запуске («расскажите, чем увлекаетесь»).
3. Неявное дополнение профиля по ходу обычных разговоров.

### 4.7 Контур для родственников
- Отдельный канал уведомлений для родных — **только тревожные сигналы**, не полный
  доступ к переписке (уважение к приватности пользователя).
- Требует явного согласия пользователя на подключение такого канала.
- Точная логика, что считается «тревогой» — открытый вопрос (см. п.8).

### 4.8 Growth-контур / вирусное распространение
Отдельная цель проекта — максимальный охват аудитории через вирусное
распространение генерируемого контента, а не только польза для пилотной группы.

- **Формат контента для распространения** (кандидаты, требуют исследования рынка
  перед приоритизацией — см. ниже): персональные открытки «доброе утро»/с
  праздником/с днём рождения для семейного чата, персональные песни, персональные
  стихи/поздравления.
- **Источник изображений**: на старте — готовые бесплатные фото/иллюстрации по
  теме, ИИ-генерация — только когда готового подходящего изображения нет
  (экономия и скорость). ~~Pinterest-подобные источники~~ — **скорректировано
  2026-07-04**: у Pinterest нет публичного API для поиска/скачивания контента,
  автоскрейпинг нарушает ToS (риск для продукта) — см. раздел 11.1, реально
  используются Pexels/Pixabay/Klipy.
- **Механика вирусности**:
  1. Контент несёт ненавязчивый бренд-элемент/ссылку «сделать такое же для своих
     близких» — приглашение получателям завести своего бота (K-фактор).
  2. Плюс тестируем активный реферал — бот сам предлагает пользователю пригласить
     родных завести себе такого же бота.
- **Доставка в семейный чат**: оба варианта на выбор пользователя — либо
  пересылает контент вручную (стандартный Telegram-forward, без интеграции), либо
  подключает бота в свой семейный чат и бот публикует туда сам.
- **Как выбираем топовые направления**: сначала — исследование рынка/трендов, какие
  форматы ИИ-контента (открытки/песни/стихи/видео-поздравления) уже вирусят
  именно в аудитории 50+ и в семейных чатах, и только потом фокусируемся на 2–3
  лучших направлениях вместо того, чтобы делать всё сразу.

---

## 5. Онбординг и настройка

Учитывая целевую аудиторию, на старте: **родственник настраивает и передаёт
пользователю уже готового бота** — сам пенсионер ничего не устанавливает и не
конфигурирует, просто открывает чат и начинает говорить.

---

## 6. Масштаб на старте

Небольшая группа пользователей (несколько человек, знакомые/близкие) — без
необходимости в сложной мультитенантной архитектуре, биллинге и публичной
регистрации. Задел на рост заложить архитектурно, но не реализовывать сразу.

---

## 7. Нефункциональные требования (предварительно)

- **Приватность и согласие**: явное согласие пользователя на хранение
  чувствительных данных (здоровье, лекарства) и на подключение родственников к
  тревожному каналу. Хранение таких данных — с шифрованием.
- **Дисклеймеры**: во всех медицинских сценариях — явное указание, что бот не
  заменяет врача.
- **Простота UX**: ноль сложных меню, всё через голос/фото, минимальное количество
  шагов для любого действия.
- **Надёжность проактивных сообщений**: учёт часового пояса пользователя, retry
  при сбоях доставки.

## 8. Открытые вопросы (зафиксированы для дальнейшего штурма/ТЗ)

1. ~~Точная логика и порог тревожного сигнала родным~~ — **РЕШЕНО (см. п.4.7):**
   базовый порог — 72 часа без ответа на чек-ин. Дополнительно: если бот улавливает
   тревожный паттерн в речи (например, жалоба на самочувствие/падение), он сам
   спрашивает пользователя вслух: «сообщить родным о вашем состоянии?». При отказе —
   не сообщает сразу, но продолжает мониторить; если паттерн повторяется — сообщает
   родным несмотря на предыдущий отказ. Content-анализ речи — экспериментальный
   модуль: сначала копим аналитику по его срабатываниям и полезности, и только потом
   решаем — развивать в отдельную важную ветку продукта или оставить в упрощённом
   виде (только по времени).
2. ~~Выбор вендоров и бюджет~~ — **РЕШЕНО:** международные вендоры (Whisper/Deepgram
   для STT, ElevenLabs/OpenAI для TTS), LLM и генерация фото/видео — через доступные
   API. Бюджет на пилотную группу — умеренный, качество важнее экономии. Дополнение:
   для декоративного/неперсонализированного контента (например, утренние открытки-
   поздравления) не обязательно каждый раз генерировать картинку ИИ — можно брать
   готовые бесплатные фото по API (Unsplash-подобные сервисы), а генерацию беречь для
   персонализированных иллюстраций и инструкций.
3. Модель оплаты/монетизации — отложена, пока фокус на функционале.
4. ~~Голосовая «личность» бота~~ — **РЕШЕНО:** на старте — выбор голоса при
   настройке родственником из нескольких готовых вариантов (муж/жен, тембр). Темп
   речи — обычный по умолчанию, с настройкой скорости под пользователя. Отдельно
   протестировать voice cloning (клон голоса близкого человека) — если качество
   хорошее и цена не топовая, рассмотреть как премиум-опцию для отдельной категории
   пользователей.
5. ~~Политика хранения данных~~ — **РЕШЕНО:** базовый профиль (имя, возраст,
   интересы) хранится бессрочно — это основа памяти и персонализации. Чувствительные
   данные (здоровье, лекарства, конкретные рецепты) — гибкий срок, например
   актуальность/устаревание записи (старый рецепт можно автоматически архивировать/
   удалять), точные правила уточнить на этапе реализации.
6. ~~Технический механизм подключения родственника~~ — **РЕШЕНО:** отдельный
   Telegram-бот-уведомитель только для родных/опекунов — чёткое разделение ролей
   (основной бот — компаньон пользователя, второй — канал тревог), проще
   масштабировать на нескольких родственников одного пользователя.
7. ~~Оформление согласия~~ — **РЕШЕНО (частично):** на пилоте среди своих —
   формальное оформление пропускаем. Для будущего продукта — планируем голосовое
   согласие от самого пользователя при онбординге. Юридическая проработка (форма
   согласия, соответствие 152-ФЗ для персональных/медицинских данных) — отдельная
   задача перед выходом за пределы тестовой группы своих.
8. ~~Список хобби-веток~~ — **РЕШЕНО:** список заранее не фиксируем. Дача/огород,
   рыбалка, пчеловодство — уже названные примеры, но реальный набор веток соберём
   по факту из онбординга пилотной группы (см. п.4.6 — визуальный выбор + анкета +
   неявное дополнение).

Все 8 открытых вопросов проработаны. Оставшиеся частично открытые моменты (не
блокируют MVP, но требуют внимания перед выходом за пределы пилота «своих»):

- Юридическое оформление согласий и соответствие 152-ФЗ при обработке персональных/
  медицинских данных (п.7).
- Точные правила автоархивации чувствительных данных (п.5).
- Оценка полезности content-анализа речи для тревожного модуля — по результатам
  накопленной аналитики на пилоте (п.1).

## 9. Growth-исследование (итоги)

Проведено 4-поточное исследование по growth-направлению (открытки, песни, стихи,
виральные механики) — полные находки в `growth-research-findings.md`. Кратко:

- **Открытки — приоритет 1**: дёшево, совпадает с уже существующей привычкой
  аудитории 50+ пересылать картинки-поздравления; подтверждённый поведенческий
  прецедент (феномен «доброе утро» в WhatsApp у пожилых пользователей в Индии) и
  сильная аналогия потенциала (Studio Ghibli-эффект в ChatGPT, +1 млн юзеров/час).
- **Голосовая озвучка стиха — усиление открытки**, не отдельный формат: голый
  текст виралится слабее визуала/аудио, но естественно ложится на голосовую
  природу продукта.
- **Песни — приоритет 2, тестировать отдельно**: технология дешёвая (~2–15
  руб/песня), но нет официального API у Suno/Udio и есть неурегулированные иски
  лейблов (на июль 2026 — Sony ещё судится и с Suno, и с Udio; решение по
  прецедентному вопросу fair use ожидается летом 2026) — слишком рискованно для
  core MVP.
- **Механика распространения — гибрид**: пассивная контент-вирусность (сам
  контент — двигатель шеринга) + мягкий deep-link-CTA на контенте + точечный тест
  активного реферала. Отраслевых бенчмарков K-фактора для этой ниши нет — считаем
  свой с нуля на пилоте.

## 10. MVP-скоуп (первая итерация)

**Объём**: средний — ядро компаньона + отобранные фото-сценарии + открытки
(growth-приоритет 1), без песен (откладываются на отдельный тест).

**Входит в MVP**:
- Голосовой диалог + память о пользователе (профиль, интересы, факты из
  разговоров).
- Проактивность: утренний/вечерний чек-ин, напоминания о лекарствах/режиме,
  поздравления с датами, детект «давно нет на связи» (72 часа — базовый порог).
- Фото-сценарии — **все четыре** из названных на старте (шире, чем изначально
  предполагал средний уровень объёма, но явно выбраны пользователем как MVP-скоуп):
  рецепты/медицинские документы, растения и уход за садом/огородом, бытовые
  документы (ЖКХ, письма), еда/продукты (состав, польза/вред).
- Открытки-поздравления (growth-приоритет 1) + голосовая озвучка стиха как
  усиление, с мягким deep-link-приглашением «сделать такое же для своих».
- Онбординг: родственник настраивает и передаёт готового бота; профиль интересов
  собирается визуальным выбором + анкетой + неявным дополнением по разговорам.
- Контур для родных: отдельный бот-уведомитель, тревога — по 72-часовому порогу
  молчания (content-анализ речи — экспериментальный модуль, не в MVP до
  накопления аналитики).

**Не входит в MVP (бэклог второй волны)**:
- Персональные ИИ-песни — отдельный тест на пилоте, не core-фича.
- Voice cloning (клонирование голоса близкого) — премиум-опция, тестировать
  отдельно.
- Активный реферал от бота («предложить пригласить родных») — точечный тест
  после наблюдения за пассивной вирусностью.
- Юридическое оформление согласий (152-ФЗ) — до выхода за пределы пилота «своих».

**Пилот**: 3–5 пользователей, запуск в течение месяца.

**Главная метрика успеха на старте**: **виральность контента** — сколько
открыток/поздравлений переслано в семейные чаты и сколько по ним перешли/завели
своего бота (свой K-фактор с нуля, поскольку отраслевых бенчмарков для этой ниши
нет). Дополнительно отслеживать регулярность использования и эмоциональный
отклик пользователя/семьи как вторичные сигналы.

## 11. Discovery-функция: «обучение у аудитории»

Отдельная подзадача поверх MVP (штурм от 2026-07-04): цель — выявить самые
массовые и полезные направления коммуникации/пользы для аудитории, чтобы стать
незаменимым и вирусным «как соцсети», а не просто набором из 4 фото-сценариев и
открыток.

**Механика**:
- Бот анонсирует широкий спектр возможностей голосового ИИ вообще (не только уже
  реализованное: открытка, совет, любая болталка) — приглашает пробовать и
  просить о любых задачах.
- На произвольную/нетипичную просьбу, которую бот ещё не умеет выполнять
  качественно, — не отказывает и не ограничивается дисклеймером, а **пытается
  ответить как есть** через свободный диалог (brain.respond уже это умеет).
- Каждый такой запрос — сигнал для продуктовой аналитики: что реально просит
  аудитория, какие направления оказываются самыми частыми/ценными.

**Аналитика запросов (важно — зафиксировано двухэтапное решение)**:
1. **Сейчас (этап 1)**: автоматическая LLM-классификация каждого входящего
   сообщения по категории запроса (доп. тег/вызов при обработке в brain.py) —
   пишется в БД сразу, с первого дня.
2. **Далее (этап 2, когда данных станет больше)**: перейти на пакетный анализ по
   расписанию (батч-обработка накопившегося лога) вместо тегирования каждого
   сообщения — дешевле по токенам, приемлемо, что результат появляется не в
   реальном времени. Дата перехода — когда объём данных/трафика оправдает
   экономию (ориентир пересмотреть после пилота).

**Отчёт о популярных направлениях** — выводить в Google Sheets, как и остальные
сводки по проекту (см. [[feedback-tables-to-google-sheets]] в памяти) — не в
терминал/чат.

**Реализовано (2026-07-04)**:
- Таблица `request_topics` + `db.log_request_topic()`/`get_request_topics()`.
- Третий tool-use `log_request_topic` в brain.py — вызывается на каждое сообщение,
  свободная метка без фиксированной таксономии; выполняется как побочный эффект
  без лишнего round-trip к API (текст ответа и тег обычно приходят в одном вызове).
- Google-таблица создана: «Добро — популярные направления запросов»,
  https://docs.google.com/spreadsheets/d/1ORrwvmx0Eumjw4eYgIGTS8vQeIL4KibFohYYLkKTIa4/edit
  (пока пустая — пилот ещё не запущен).

**Техническое ограничение** (важно на будущее): доступный Google Drive MCP не
умеет редактировать ячейки существующей таблицы — только создавать новый файл
целиком (CSV→Sheets, см. технику в [[finmodel-fabplast-pnl]]). Значит обновление
отчёта — это создание НОВОЙ версии таблицы (или ручное копирование пользователем),
не автоматический live-апдейт. На пилоте с 3-5 пользователями это нормально:
отчёт будем генерировать по запросу («покажи популярные направления»), читая
`request_topics` из `data/dobro.sqlite3` напрямую с диска сервера.

**Обнаруженный попутно пробел (не исправлялся, требует решения)**: просьба
открыткой в свободной речи («пришли открытку с добрым утром») сейчас НЕ
триггерит `postcards.generate_postcard` — это работает только по явной команде
`/postcard`. Естественный fix — четвёртый tool-use в brain.py (`send_postcard`),
который дёргает postcards.py и telegram_client.py как побочный эффект, аналогично
remember_fact/schedule_reminder.

### 11.1 Персонализация открыток (штурм 2026-07-04)

Идеальная открытка разная для каждого человека (кошечки одному, мультяшная
фигурка другому) — решено определять стиль и сюжет автоматически из уже
известных интересов (`memory_facts`, категория `interest`), а не отдельным
UI-вопросом. Стиль картинки (фото/иллюстрация/анимация) выбирает модель под
конкретный повод, а не фиксируется на пользователя раз и навсегда.

**Реализовано**:
- Один forced tool-use вызов (`ANTHROPIC_FAST_MODEL`) сразу отдаёт стиль,
  поисковый запрос по интересам и текст стиха — экономия на API-вызовах.
- Источники изображений с откатом по цепочке: **Pexels** (реальные фото) →
  **Pixabay** (иллюстрации/векторы — закрывает «мультяшную» стилистику, тоже
  фото как запасной вариант) → **Klipy** (анимация/GIF, только когда стиль
  `animation`) → `picsum.photos` (generic-фото, последний откат без ключа).
- Текст стиха накладывается на статичную картинку через Pillow
  (DejaVuSans-Bold, кириллица подтверждена) — полупрозрачная плашка снизу,
  перенос строк по ширине. Для анимаций текст остаётся отдельной подписью
  (наложение на GIF не делали — не нужно для MVP).
- `telegram_client.send_animation()` — для GIF/MP4-открыток.

**Pinterest исключён** (см. корректировку в разделе про growth-контент выше):
нет публичного API для поиска/скачивания, автоскрейпинг нарушает ToS — риск
для продукта. Все три реальных источника (Pexels/Pixabay/Klipy) требуют
бесплатной регистрации и отдельного API-ключа — без них система работает,
просто откатывается на generic-фото с picsum.photos.

**РЕШЕНО 2026-07-04**: все три ключа (Pexels/Pixabay/Klipy) получены и
подтверждены живыми тестами. По пути обнаружены и исправлены два реальных
бага: (1) все источники требуют браузероподобный User-Agent — иначе
Cloudflare блокирует запрос как бота (403 "error code: 1010"); (2) ответ
Klipy на один уровень вложенности глубже, чем изначально предполагалось —
`{"data": {"data": [...]}}`, не `{"data": [...]}`. Оба фикса — в `postcards.py`.
Сквозные тесты дали реальные тематические открытки (фото рыбака под интерес
«рыбалка», иллюстрация с котятами под «вышивку и котят», валидный GIF по
«birthday celebration»).

### 11.2 Гипотеза: изображения с Pinterest vs банки-конкуренты (штурм 2026-07-04)

**Наблюдение пользователя**: изображения по запросам вроде «доброе утро» на
Pinterest нравятся заметно больше, чем то, что скорее всего выдадут Pexels/
Pixabay по тому же запросу.

**Вероятная причина**: Pinterest для таких запросов в основном показывает не
обычные фотографии, а **готовые графические открытки** — каллиграфия, цветочные
рамки, пастельная стилистика, специально сделанные дизайнерами для расшаривания.
Это принципиально другая категория контента, чем «сырая» стоковая фотография
или иллюстрация из Pexels/Pixabay — прямое сравнение будет нечестным, пока не
проверим именно этот класс контента.

**План проверки** (не реализовывать сейчас — зафиксировать как гипотезу для
теста после получения ключей):
1. Как появятся ключи Pexels/Pixabay — вытянуть 5-10 примеров по запросам
   «доброе утро», «день рождения», «природа» и оценить визуально, насколько
   они близки по ощущению к тому, что нравится на Pinterest.
2. Если разрыв в качестве подтвердится — проверить **Freepik** (готовые
   дизайнерские шаблоны открыток, есть API) как альтернативный источник именно
   такого класса контента, и/или сильнее опереться на ИИ-генерацию именно под
   эстетику «открытка с каллиграфией», а не искать готовое.

**Предложенный пользователем промежуточный вариант**: не дёргать банк
изображений живым запросом на каждую открытку, а периодически (не в реальном
времени) собирать/обновлять свою локальную базу «хороших»/трендовых картинок и
отдавать открытки из неё — быстрее и не зависит от лимитов внешних API.
Технически разумно, НО:

**Критическая оговорка — это не снимает риск авторских прав.** Проблема с
Pinterest не в том, что запрос идёт «вживую» при каждой открытке, а в том, что
Pinterest агрегирует чужой контент, репостнутый пользователями со всего
интернета, — источник и лицензия конкретного пина обычно неизвестны и не
проверяемы. Собрать «базу хороших картинок с Pinterest» batch-способом —
такой же юридический риск, как скрейпить их живьём, если только:
(а) не проверять лицензию каждого изображения индивидуально (нереалистично при
масштабе), либо (б) собирать локальную базу ТОЛЬКО из легально лицензированных
источников (Pexels/Pixabay/Freepik — тогда «предпросчитанная база» становится
просто кэшем/оптимизацией их API, без юридической проблемы), либо (в) генерировать
нужную эстетику через ИИ (полностью оригинальный контент, риска нет).

**Вывод**: сама идея «своя предпросчитанная база трендовых картинок» — хорошая
оптимизация (независимо от источника), но источником для нёе должны быть
легальные банки/ИИ-генерация, а не Pinterest. Дальнейший шаг — сравнение
качества (шаг 1 плана выше), затем решение, нужна ли вообще локальная база на
этапе пилота (3-5 пользователей — вряд ли упрёмся в лимиты Pexels/Pixabay).

**РЕЗУЛЬТАТЫ СРАВНЕНИЯ (2026-07-04, два раунда, тема «доброе утро» и «ангелы»)**:
- Раунд 1 (генерик «доброе утро»): Pexels (фото) / Pixabay (иллюстрация) /
  Klipy (анимация) — **Klipy лучший**.
- Раунд 2 (тема «ангелы», по прямой обратной связи пользователя — аудитории
  заходит именно этот мотив): ИИ-генерация (Nano Banana 2, 3 варианта в стиле
  винтажной открытки) + Klipy + Pixabay — **Klipy снова лучший, ИИ-генерация
  тоже достойна**. Статичные фото/иллюстрации из Pexels/Pixabay — стабильно
  слабее по обеим темам.
- **Вывод**: подтверждает гипотезу выше — Pinterest-like ощущение ближе к
  анимации/дизайнерской иллюстрации, чем к сырому стоку. **Klipy — основной
  источник**, ИИ-генерация — сильный второй вариант.
- **Важное техническое ограничение**: ИИ-варианты в этом тесте сделаны через
  Higgsfield/MCP-доступ ЭТОЙ сессии Claude Code — сам бот (`postcards.py`,
  автономный Python-процесс в Docker) **не может** вызывать этот инструмент,
  у него нет к нему доступа. Чтобы ИИ-генерация была доступна боту в проде,
  нужна отдельная публичная API-интеграция (с собственным ключом) — это не
  сделано, требует отдельного решения/исследования (см. раздел 12).

### 11.3 Первый запуск и онбординг нового пользователя (штурм 2026-07-04)

Исследованы лучшие практики виральных онбордингов (time-to-value, момент для
приглашения, паттерны caregiving-приложений, AI-онбординг для новичков) —
источники: growth-статьи про aha-moment/activation, кейсы Dropbox/Uber
(реферал в момент пика удовлетворения, не при регистрации), caregiving-апps
(«держите онбординг крошечным»), NN/g про онбординг ИИ-новичков (общие примеры
возможностей лучше узких для формирования правильной ментальной модели).

**Решения по итогам штурма**:
- Первое сообщение после `/start` — тёплое приветствие БЕЗ автогенерации
  открытки (не быть навязчивым); открытка — когда пользователь сам попросит
  (уже работает через `send_postcard` tool-use, раздел ранее в этом документе).
- Приветствие сразу широко перечисляет возможности (открытки, советы по
  здоровью/дому, разбор фото/документов, сад/огород, просто разговор) — не
  постепенное раскрытие. Подтверждает discovery-философию (раздел 11):
  общие примеры лучше узких для формирования у новичка правильного
  представления о диапазоне возможностей.
- **Второй вирусный контур (онбординг родственника → предложить завести бота
  для другого родственника) — сознательно НЕ делаем сейчас.** Фокус пилота —
  только контур через открытки самого пожилого пользователя; расширять на
  родственника как отдельный канал роста — по факту результатов пилота.
- Момент приглашения (deep-link в подписи открытки) остаётся привязан к
  моменту получения ценности (открытки), а не к регистрации — это уже
  совпадает с рекомендацией из исследования (Dropbox/Uber паттерн).

**Реализовано**: текст приветствия в `main.py::_handle_start` обновлён —
сразу перечисляет широкий спектр возможностей.

# «Добро» — база знаний для совместной работы

Голосовой ИИ-компаньон в Telegram для аудитории 50+/пенсионеров: общение и забота,
реальная польза по здоровью/быту (напоминания, разбор рецептов и документов, сад/
огород), доступ к возможностям ИИ через голос и фото. Бот проактивен — сам пишет
первым (чек-ины, напоминания, поздравления). Growth-цель — вирусное распространение
персональных открыток в семейные чаты.

Здесь собран набор из **5 документов**, которые команда (владелец проекта + 2
разработчика) использует для совместной работы над проектом.

---

## ⚠️ О расположении этих документов

Эти файлы описывают **отдельный проект «Добро»** (Python-бот в Telegram, целевой
репозиторий `DeviceIngineering/dobro`). Они сложены сюда, в репозиторий `petai`
(Next.js), потому что на момент фиксации это был единственный доступный
репозиторий с правом записи, а репозитория `dobro` в области видимости ещё нет.

Что важно знать команде:

- **Код бота** (`bot/brain.py`, `db.py`, `postcards.py`, `scheduler.py`,
  `alert_bot/`, `tests/`, Docker и т.д.) здесь **не лежит** — он живёт в исходном
  монорепо `ClaudeProjects` и переезжает в собственный репозиторий по плану из
  [`MIGRATION_TO_GITHUB.md`](./MIGRATION_TO_GITHUB.md).
- **Инфраструктуру совместной работы** (`.github/workflows/ci.yml`, `deploy.yml`,
  `rollback.yml`, `PULL_REQUEST_TEMPLATE.md`, issue-шаблоны, `CODEOWNERS`, защиту
  ветки `main`, GitHub Environment `production`, секреты) осознанно **не создавали
  в `petai`**: она Python-ориентирована и в чужом Next.js-репозитории либо ломает
  CI, либо меняет поведение GitHub (не те ревьюеры, ложные проверки). Её место —
  в выделенном репозитории `dobro`, и создаётся она там по шагам из
  `MIGRATION_TO_GITHUB.md` (шаг 3) и `CONTRIBUTING.md` (разделы 4, 6, 7).

Как только репозиторий `dobro` появится в области видимости — этот набор
переносится туда как есть, и уже там разворачивается `.github/`-каркас.

## Модель совместной работы — кратко (полностью в `CONTRIBUTING.md`)

- **Команда:** владелец проекта + Разработчик A + Разработчик B. На проде — живые
  пилотные пользователи 50+, поэтому **надёжность мержа в `main` важнее скорости**.
- **Ветвление:** trunk-based. `main` всегда деплоится, прямой push запрещён. Одна
  задача → одна ветка (`feature/<кратко>` или `fix/<кратко>`) → один PR → **squash
  merge** (один PR = один коммит в истории `main`).
- **Ревью:** разработчики ревьюят друг друга (мин. 1 approval), владелец
  подключается на рискованных/breaking изменениях. Единственная линия защиты перед
  проверенным продом — поэтому компенсируем высоким покрытием тестами.
- **Тесты — главная линия защиты:** каждый fix обязан добавить регрессионный тест;
  каждая новая логика — покрытие тестами; внешние сервисы мокаются; порог покрытия
  — «храповик» (`COVERAGE_MIN`, старт `25`, поднимать вручную осознанным решением
  команды; цель на критических модулях — 60–70%).
- **Окружения:** общего staging нет — у каждого разработчика свой личный тестовый
  Telegram-бот и свой `.env`. Секреты — никогда не в git/чате/PR. CI не зовёт
  реальные внешние API — только моки.
- **CI/CD:** `ci.yml` (lint → тесты с покрытием → сборка Docker) на каждый PR и на
  `main`; деплой и откат — **ручные** (`workflow_dispatch` + approval через GitHub
  Environment `production`), с бэкапом БД и `prod-<дата>`-тегом как точкой отката.
- **Документация:** продуктовая концепция меняется через PR; технические решения —
  в `CHANGELOG.md` (Keep a Changelog) при мерже, если менялось видимое пользователю
  поведение или инфраструктура.

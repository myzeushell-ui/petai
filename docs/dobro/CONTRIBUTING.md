# Как мы разрабатываем «Добро» вместе

Команда: 3 человека (владелец проекта + 2 разработчика). На проде — живые
пилотные пользователи 50+/пенсионеры. Главный принцип этого документа:
**сломанный бот на проде бьёт по реальным людям**, поэтому скорость разработки
всегда уступает надёжности мержа в `main`.

Если что-то в этом документе противоречит здравому смыслу в конкретной
ситуации — сначала обсудить с командой, потом менять документ, а не тихо
игнорировать правило.

---

## 1. Роли

- **Владелец проекта** — держит концепцию (`concept-tz-draft.md`, `docs/`),
  утверждает изменения продукта/UX, ревьюит инфраструктуру и общую БД-схему
  (`bot/db_schema.sql`, `bot/brain.py` — общая критическая поверхность).
- **Разработчик A / Разработчик B** — распределение по модулям в
  `.github/CODEOWNERS` (заполнить реальными GitHub-логинами при подключении).
  Разделение мягкое — это подсказка, кого звать ревьюером первым, не запрет
  трогать чужой модуль.

## 2. Режим штурма ТЗ

Перед реализацией новой идеи (не бага, а именно новой функции/изменения
концепции) — сначала штурмим и доводим ТЗ до готовности в обсуждении с
владельцем, потом реализуем. Не начинать кодить по идее, брошенной вскользь.
Шаблон issue `feature.md` содержит чекбокс именно на этот случай.

## 3. Ветвление

Модель — trunk-based, без долгоживущего `develop` (staging-окружения нет, см.
раздел 5, поэтому `develop`-ветка была бы лишней сложностью без места, куда её
деплоить):

- `main` — всегда в состоянии, которое можно задеплоить. Прямой push запрещён
  (защита ветки, см. раздел 4).
- Одна задача — одна ветка — один PR. Именование: `feature/<кратко>` или
  `fix/<кратко>`. Ветки короткоживущие (дни, не недели) — чем дольше живёт
  ветка, тем больнее мерж-конфликты.
- Мерж в `main` — **squash merge**. Один PR = один коммит в истории `main`.
  Это даёт: чистую историю, и возможность откатить одну фичу одним
  `git revert` при необходимости.

## 4. Ревью и защита `main`

Договорённость команды: **разработчики ревьюят друг друга**, владелец
подключается по запросу или на рискованных/breaking изменениях (см. чекбокс
"breaking" в `PULL_REQUEST_TEMPLATE.md`). Так как это единственная линия
защиты перед проверенным продом, компенсируем это высоким покрытием тестами
(раздел 6) — ревью проверяет логику и риски, тесты проверяют, что ничего
старого не сломалось.

**Настройки защиты ветки `main` на GitHub** (Settings → Branches → добавить
правило для `main`; сделать вручную при создании репозитория — API-доступа
для этого сейчас нет, см. `docs/MIGRATION_TO_GITHUB.md`):
- Require a pull request before merging — включено, direct push запрещён.
- Require approvals — минимум 1.
- Dismiss stale approvals on new commits — включено (не одобрять старый диф).
- Require status checks to pass — все джобы `ci.yml` (lint, test, docker-build).
- Require branches to be up to date before merging — включено.
- Include administrators — включено (владелец тоже мержит только через PR).
- Запретить force-push и удаление ветки `main`.

## 5. Окружения и тестирование (без общего staging)

Решение команды: отдельного постоянного staging-бота **нет** — усложняет
инфраструктуру ради пилота на 3-5 пользователей. Вместо этого:

- **У каждого разработчика — свой личный тестовый Telegram-бот** (заводится
  через @BotFather за 1 минуту, бесплатно, свой токен) и свой `.env` locally
  (см. `.env.example`/`.env.alert.example` — какие переменные нужны).
  Продовый `.env` существует только на сервере и никогда не покидает его.
- Тестируем на личном боте перед PR, описываем сценарий в чек-листе PR
  ("Как протестировано").
- API-ключи для разработки (Anthropic/Deepgram/ElevenLabs/Pexels/Pixabay/
  Klipy) — либо свои бесплатные ключи (Pexels/Pixabay/Klipy это позволяют),
  либо общий dev-ключ через менеджер паролей команды. **Никогда не в git, не
  в чате, не в PR-описании.**
- CI (раздел 6) не должен звать реальные внешние API вообще — только моки.
  Значит секреты в GitHub Actions для CI не нужны в принципе.

## 6. Тесты и покрытие — главная линия защиты

Поскольку общего ревью-гейта владельца нет, тестовое покрытие — это то, что
реально не даёт сломать проверенный функционал. Правила:

- **Каждый PR с fix — обязан добавить регрессионный тест**, который бы поймал
  этот баг (чек-лист PR это явно требует). У нас уже есть прецедент —
  `tests/test_voice_kind.py` ловит именно тот баг, который был на проде
  (голосовые логировались как `kind='text'`). Так же оформлять на будущее:
  баг с годом в `expires_at`, баг с одинаковыми повторными открытками и т.д.
  задним числом — по мере сил, не блокирующе.
- **Каждый PR с новой логикой — обязан покрыть её тестами.** Все внешние
  сервисы мокаются (пример — `tests/test_voice_kind.py`,
  `tests/conftest.py`).
- **Порог покрытия — "храповик" (ratchet), не фиксированное число.** CI
  проверяет `--cov-fail-under`, значение берётся из переменной репозитория
  `COVERAGE_MIN` (Settings → Secrets and variables → Actions → Variables).
  **Стартовое значение — `25`** (реальное покрытие на момент написания — 26%,
  один тест на `bot/`). По мере добавления тестов — поднимать `COVERAGE_MIN`
  вручную (не автоматически, чтобы это было осознанным решением команды, а не
  тихим правилом, которое никто не помнит). Цель на первые несколько недель
  командной работы — дойти до 60-70% на критических модулях (`brain.py`,
  `postcards.py`, `scheduler.py`, `db.py`) — именно они управляют памятью,
  напоминаниями и открытками, то есть проверенным функционалом пилота.
- Локальный прогон:
  ```
  pip install -r requirements-dev.txt
  pytest tests/ -v --cov=bot --cov=alert_bot --cov-report=term-missing
  black bot/ alert_bot/ tests/ && isort --profile black bot/ alert_bot/ tests/
  flake8 bot/ alert_bot/ tests/
  ```

## 7. CI/CD

- **`.github/workflows/ci.yml`** — на каждый push в PR и на `main`: линт
  (flake8/black/isort) → тесты с покрытием → сборка обоих Docker-образов.
  Все три джобы должны быть зелёными для мержа (раздел 4).
- **`.github/workflows/deploy.yml`** — **деплой в прод НЕ автоматический** на
  мерж в `main`. Осознанный запуск вручную (`workflow_dispatch`) с
  подтверждением + approval через GitHub Environment `production`. Перед
  обновлением кода: бэкап `data/dobro.sqlite3` и git-тег `prod-<дата>` на
  текущем состоянии (точка отката). После деплоя — smoke-check, что оба
  контейнера `healthy` (используем существующие Docker HEALTHCHECK).
- **`.github/workflows/rollback.yml`** — быстрый откат на предыдущий
  `prod-*`-тег, тоже ручной запуск. Использовать сразу, если после деплоя
  что-то не так, не разбираясь на живую — сначала откатить, потом чинить в
  ветке.

Требуемые секреты в GitHub (Settings → Secrets and variables → Actions):
`DEPLOY_HOST`, `DEPLOY_USER`, `DEPLOY_SSH_KEY`, `DEPLOY_PATH`. Рекомендуется
отдельный SSH-ключ только для деплоя (не переиспользовать личные ключи
разработчиков), при возможности — ограниченный `command=` в
`authorized_keys` на сервере.

## 8. Данные и схема БД

- Любое изменение `bot/db_schema.sql` — ревьюит владелец или оба разработчика
  (см. CODEOWNERS), потому что затрагивает общую боевую БД пилота.
- Миграции — идемпотентные (`CREATE TABLE IF NOT EXISTS` и подобное, как
  сейчас в `db.init_db()`), без ORM-фреймворка миграций — оправдано для
  пилота на 3-5 пользователей, пересмотреть при росте.
- `deploy.yml` бэкапит `data/dobro.sqlite3` перед каждым обновлением — при
  проблеме со схемой можно восстановить файл вручную с сервера.

## 9. Документация

- Продуктовая концепция — `concept-tz-draft.md`, её эволюция — в
  `docs/evolution/` (см. `docs/README.md`). Меняется через PR, не напрямую.
- Технические решения по ходу работы — фиксировать в `CHANGELOG.md`
  (формат Keep a Changelog) при мерже PR, если менялось поведение бота
  видимое пользователю или инфраструктура.

## 10. Онбординг нового разработчика — чек-лист

1. Доступ в репозиторий `DeviceIngineering/dobro` на GitHub.
2. Склонировать, `pip install -r requirements-dev.txt`.
3. Завести личного тестового бота через @BotFather, получить токен.
4. Скопировать `.env.example` → `.env`, заполнить своим TG-токеном и рабочими
   dev-ключами (спросить у команды, где взять — не в git).
5. Прогнать тесты локально (раздел 6) — должны быть зелёными на чистом
   клоне.
6. Прочитать `concept-tz-draft.md` и `docs/README.md` — понять, где сейчас
   продукт и что решено/не решено.
7. Взять issue, завести ветку `feature/...` или `fix/...`, работать по
   разделам 3-6 этого документа.
